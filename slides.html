What is Bazar-Trace?

  • A Single Page Application (SPA) built purely using raw HTML5, Vanilla CSS, and modular ES6 JavaScript.
  • Implements an Offline-First Engine to guarantee sales logging during internet drops.
  • Runs local product lookups and scans via hardware integrations.
  • Secured with role-based access controllers.
Core Performance Metrics

✓ Uptime: 100% Offline Resilience

✓ Size: Minimal static bundle shell

✓ Client Cache: Asynchronous IndexedDB

✓ Relational Core: Oracle DB Integrations

Motivation & Problem Statement

  1. Unstable Connectivity:
    Retail shops in Bangladesh frequently lose internet access, disabling normal cloud systems.
  2. Heavy Software Costs:
    Small businesses cannot afford costly licenses or local server setups.
  3. Library Dependencies Constraint:
    Proving that critical sync logic can be built without heavy framework layers (React/Next.js).
The Cost of Offline Outages

"If a store loses connection for even 30 minutes during rush hour, inventory counts mismatch, receipts cannot compile, and checkout queues stall."

Project Goals

  • Zero Latency Checkout: Enable immediate invoice inputs and product queries local to the machine.
  • FIFO Sync Queue: Queue database updates and drain automatically when networks restore.
  • Hardware Scanning: Utilize camera permissions for instant product identification.
  • Educational Focus: Maximize understanding of raw browser web APIs (Service Worker, Web Audio, Canvas).

Database Schema Configuration


CREATE TABLE products (
  id NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  sku VARCHAR2(50) UNIQUE NOT NULL,
  barcode VARCHAR2(100),
  name VARCHAR2(150) NOT NULL,
  stock NUMBER DEFAULT 0 NOT NULL,
  cost_price NUMBER(10,2) NOT NULL,
  sell_price NUMBER(10,2) NOT NULL,
  low_stock_threshold NUMBER DEFAULT 5 NOT NULL,
  expiry_date DATE
);

CREATE TABLE transactions (
  id NUMBER GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
  client_txn_id VARCHAR2(50) UNIQUE NOT NULL,
  product_id NUMBER REFERENCES products(id),
  user_id NUMBER REFERENCES users(id),
  txn_type VARCHAR2(10) CHECK (txn_type IN ('IN', 'OUT')),
  quantity NUMBER NOT NULL,
  unit_price NUMBER(10,2) NOT NULL,
  occurred_at TIMESTAMP DEFAULT SYSTIMESTAMP
);

SWOT Analysis

Strengths (S)

• 100% offline uptime guarantee

• Zero framework overhead (high speed)

• Idempotent queue sync checks

Weaknesses (W)

• Local DB updates rely on sync triggers

• Camera scan depends on browser support

Opportunities (O)

• Expand into multi-terminal retail cash registers

• Direct thermal receipt printers binding

Threats (T)

• Browser cache wipes delete un-synced queue items

• HTTPS restrictions for getUserMedia APIs
