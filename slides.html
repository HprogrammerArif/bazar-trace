📦 What is Bazar-Trace?

  • Vanilla Single Page Application — An SPA built purely with raw ES Modules, HTML5, and CSS variables (no React, Next.js, or Tailwind CSS).
  • Offline-First Sync Engine — Continues recording transactions and looking up products even when the internet drops completely.
  • Role-Based Security — Strict separation of admin management and staff registry options.
  • Mobile-First Design — Optimized for low-end devices and simple mobile browsers.
2 / 15

💡 Motivation & Problem Statement

  1. Internet Instability — Retail shops in Bangladesh frequently suffer from network drops, rendering standard cloud tools unusable.
  2. High Overhead Costs — Small store owners cannot afford expensive desktop setups or heavy software licenses.
  3. UI/UX Clutter — Existing ERP and inventory portals are too complex for non-technical cashiers.
  4. Academic Constraint — The challenge of building sophisticated offline sync and real-time canvas charting without heavy third-party libraries.
3 / 15

🎯 Project Goals

  1. Zero Downtime — Enable cashiers to record Sales (OUT) and Stock received (IN) with zero latency during offline states.
  2. Conflict-Free Auto Synchronization — Replay offline mutations in strict FIFO order on network recovery, handling sync retries idempotently.
  3. Hardware Integrations — Access native camera modules for barcode scanning and trigger synthesized audio scan beeps.
  4. Resource Constraints Compliance — Build the entire system under strict university guidelines (Vanilla CSS, raw JS, Oracle DB).

7 / 15

📊 Non-Functional Requirements

  • Performance — UI response time under 50ms; static bundle load under 200ms.
  • Availability — 100% operational offline. App shell cached, allowing disconnections at any stage.
  • Reliability — FIFO sync queue prevents duplicate inserts via unique transaction UUIDs (idempotency).
  • Security — Cryptographic Bcrypt hashing for credentials; token-based JWT route authentication.
  • Usability — Minimalist layouts, clear alert badges, automatic input formatting.

💾 Oracle SQL Schema

CREATE TABLE users ( id NUMBER GENERATED ALWAYS AS IDENTITY PRIMARY KEY, email VARCHAR2(160) UNIQUE NOT NULL, password_hash VARCHAR2(255) NOT NULL, full_name VARCHAR2(120) NOT NULL, role VARCHAR2(20) DEFAULT 'STAFF' CHECK (role IN ('ADMIN','STAFF')), is_active NUMBER(1) DEFAULT 1 CHECK (is_active IN (0,1)) ); CREATE TABLE products ( id NUMBER GENERATED ALWAYS AS IDENTITY PRIMARY KEY, sku VARCHAR2(64) UNIQUE NOT NULL, barcode VARCHAR2(64), name VARCHAR2(200) NOT NULL, cost_price NUMBER(12,2) NOT NULL, sell_price NUMBER(12,2) NOT NULL, low_stock_threshold NUMBER DEFAULT 5, expiry_date DATE, created_by NUMBER REFERENCES users(id) ); CREATE TABLE transactions ( id NUMBER GENERATED ALWAYS AS IDENTITY PRIMARY KEY, client_txn_id VARCHAR2(40) UNIQUE NOT NULL, -- idempotency key product_id NUMBER REFERENCES products(id), user_id NUMBER REFERENCES users(id), txn_type VARCHAR2(10) CHECK (txn_type IN ('IN','OUT')), quantity NUMBER(12,3) NOT NULL, unit_price NUMBER(10,2) NOT NULL, occurred_at TIMESTAMP DEFAULT SYSTIMESTAMP );

📈 SWOT Analysis

Strengths

  • • 100% offline uptime via IndexedDB + Service Workers
  • • Ultralight — zero JavaScript framework overhead
  • • Conflict-free transactions via client UUID idempotency

Weaknesses

  • • Database updates rely on client sync triggers
  • • Camera barcode detection needs modern browser

Opportunities

  • • Multi-terminal retail POS expansion
  • • Local hardware receipt printer integration

Threats

  • • Device cache-clear could affect unsynced data
  • • Evolving browser security for camera feeds
